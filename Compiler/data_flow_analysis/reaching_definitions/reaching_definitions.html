    到达定值

    我们可能希望找出在某个程序点上一个变量可能有哪些值，以及这些值可能在哪里定值。考虑下面这段代码：

    d1:a←1d2:if read() <= 0 goto d6d3:b←ad4:a←6d5:goto d2d6:other instructions \begin{aligned} &d_1: \quad a \leftarrow 1 \\ &d_2: \quad \text{if read() <= 0 goto } d_6 \\ &d_3: \quad b \leftarrow a \\ &d_4: \quad a \leftarrow 6 \\ &d_5: \quad \text{goto } d_2 \\ &d_6: \quad \text{other instructions} \\ \end{aligned} ​d1​:a←1d2​:if read() <= 0 goto d6​d3​:b←ad4​:a←6d5​:goto d2​d6​:other instructions​

    我们可能对程序点( d6d_6d6​ )上的所有程序状态进行总结：aaa 的值总是 1,6{1, 6}1,6 中的一个，而它由 d1,d4{d_1, d_4}d1​,d4​ 中的一个定值。可能沿着某条路径到达某个程序点的定值称为到达定值(reaching definition)。

    到达定值是最常见和有用的数据流模式之一。只要知道当控制到达程序中每个点的时候，每个变量 xxx 可能在程序中的哪些地方被定值，我们就可以确定很多有关 xxx 的性质。比如，一个编译器能够根据到达定值信息知道 xxx 在点 ppp 上的值是否为常量，而如果 xxx 在点 ppp 上被使用，则调试器可以指出 xxx 是否未经定值就被使用。

    未定值先使用

    我们可以在流图的入口处对每个变量 xxx 引入一个哑定值(dummy definition)。如果 xxx 的哑定值到达了一个可能使用 xxx 的程序点 ppp，那么 xxx 就可能在定值之前被使用。请注意，我们永远不能绝对肯定这个程序包含一个错误。因为有可能存在某种原因使得到达 ppp 点而没有真正对 xxx 赋值的路径实际上并不存在。这个原因可能涉及复杂的逻辑问题。考虑以下程序片段：

    int x;		// x_dummy
if (不透明谓词（一个复杂的数论真命题）) {
    x = 10; // x_1
}
printf("%d\n", x); // x_2

    由于 xdummyx_{dummy}xdummy​ 和 x1x_1x1​ 的值一定不同。所以分析器必须假设 xdummyx_{dummy}xdummy​ 和 x1{x_1}x1​ 都可能到达第五行代码。但实际上 xxx 未定义的情况永远不会发生，但是编译器根据分析结果将会报告 xxx 可能存在未定义被使用。这个反例展示了静态分析的根本局限性，它能识别所有可能的错误路径，但无法确定哪些路径会在实际执行中发生。这就是为什么到达定值分析只能给出“可能存在未定义使用”的警告，而非确定性结论。

    到达定值的传递方程

    我们为到达定值问题设置约束。首先检查单个语句的细节。考虑一个定值：

    d:u←v θ w d: \quad u \leftarrow v \ \theta \ w d:u←v θ w

    在这里 ←\leftarrow← 代表一个赋值操作，θ\thetaθ 表示一个一般性的二元运算符。

    这个语句“生成”了一个变量 uuu 的定值 ddd，并“杀死”了程序中其他对 uuu 的定值，而进入这个语句的其他定值都没有收到影响。因此，定值 ddd 的传递函数可以被表示为

    fd(x)=gend ∪ (x−killd) f_d(x) = gen_d \ \cup \ (x - kill_d) fd​(x)=gend​ ∪ (x−killd​)

    其中 gend={d}gen_d = \{ d \}gend​={d}，即由这个语句生成的定值的集合，而 killdkill_dkilld​ 是程序中对所有其他 uuu 的定值。一个基本块的传递函数可以通过把它包含的所有语句的传递函数组合起来而构造得到。假设有两个函数 f1(x)=gen1 ∪ (x−kill1)f_1(x) = gen_1 \ \cup \ (x - kill_1)f1​(x)=gen1​ ∪ (x−kill1​) 和 f2(x)=gen2 ∪ (x−kill2)f_2(x) = gen_2 \ \cup \ (x - kill_2)f2​(x)=gen2​ ∪ (x−kill2​)。那么

    f2(f1(x))=gen2 ∪ (gen1 ∪ (x−kill1)−kill2)=(gen2 ∪ (gen1−kill2)) ∪ (x−(kill1 ∪kill2)) \begin{aligned} f_2(f_1(x)) &= gen_2 \ \cup \ (gen_1 \ \cup \ (x - kill_1) - kill_2 ) \\ &= (gen_2 \ \cup \ (gen_1 - kill_2)) \ \cup \ (x - (kill_1 \ \cup kill_2 )) \end{aligned} f2​(f1​(x))​=gen2​ ∪ (gen1​ ∪ (x−kill1​)−kill2​)=(gen2​ ∪ (gen1​−kill2​)) ∪ (x−(kill1​ ∪kill2​))​

    fB(x)=genB ∪ (x−killB) f_B(x) = gen_B \ \cup \ (x - kill_B) fB​(x)=genB​ ∪ (x−killB​)

    其中

    killB=kill1 ∪ kill2 ∪ ... ∪killn kill_B = kill_1 \ \cup \ kill_2 \ \cup \ ... \ \cup kill_n killB​=kill1​ ∪ kill2​ ∪ ... ∪killn​

    genB=genn ∪ (genn−1−killn) ∪ (genn−2−killn−1−killn) ∪ ... ∪ (gen1−kill2−kill3−...−killn) gen_B = gen_n \ \cup \ (gen_{n-1} - kill_n) \ \cup \ (gen_{n-2} - kill_{n - 1} - kill_n) \ \cup \ \\ ... \ \cup \ (gen_1 - kill_2 - kill_3 - ... - kill_n) genB​=genn​ ∪ (genn−1​−killn​) ∪ (genn−2​−killn−1​−killn​) ∪ ... ∪ (gen1​−kill2​−kill3​−...−killn​)

    和单个语句一样，一个基本块也会生成一个定值集合并杀死一个定值集合。集合 genBgen_BgenB​ 包含了所有在紧靠基本块之后的点上“可见”的该基本块中的定值——我们把他们成为“向下可见”的。在一个基本块中，一个定值是向下可见的，仅当它没有被同一个基本块中较后的对同一变量的定值“杀死”。一个基本块的 killkillkill 集就是所有被块中各个语句杀死的定值的集合。

    一个定值可能同时出现在基本块的 gengengen 集和 killkillkill 集中。在这种情况下，该定值会被这个基本块生成，即优先考虑该定值是否在 gengengen 集中。这是因为在 gen−killgen-killgen−kill 形式中，killkillkill 集会在 gengengen 集之前被使用。

    初始值和安全值

    我们知道到达定值的格结构为 幂格集，其中 ⊤\top⊤ 是 {∅}\{ \empty \}{∅} ，表示 “无定义到达”，最乐观假设。⊥\bot⊥ 是全集 DDD，表示“所有定义都可能到达”，这是最保守的假设，也是我们优化时最不希望的结果。

    到达定值的安全值可以设置为 ⊤\top⊤ 或者 ⊥\bot⊥。根据单调性和不动点定理：到达定值的传递函数是单调的，且格是有限高度的，迭代算法必然收敛到一个不动点。在到达定值中，无论从 ⊤\top⊤ 还是 ⊥\bot⊥ 开始迭代，最终都会收敛到同一个不动点。这是因为，从 ⊤\top⊤ 开始（空集）：通过传递函数逐步添加定义（genBgen_BgenB​），并集合并路径，最终覆盖所有可达定义。收敛速度通常更快（初始值接近最终解），依据标准最小不动点理论。而从 ⊥\bot⊥ 开始（全集）：通过传递函数移除不可达定义（killBkill_BkillB​），并集保留可达定义，最终收敛到相同结果。收敛速度可能更慢（需多轮迭代移除多余定义），依据最大不动点。
