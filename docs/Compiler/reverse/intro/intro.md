# 逆向分析常用技术介绍

## 理论工具

### 控制依赖图

在程序分析和编译优化领域，**控制依赖图（CDG）**、**数据依赖图（DDG）** 和**支配图（Dominator Graph）** 是构建程序依赖关系的基础数据结构。它们均以 **控制流图（CFG）** 为输入，但刻画了不同维度的依赖信息。以下从定义、构造算法及三者区别进行详细讲解。

---

## 📌 1. 支配图（Dominator Graph）

### 1.1 定义

在 CFG 中，节点 \(d\) **支配（dominate）** 节点 \(n\)（记作 \(d \text{ dom } n\)），当且仅当从**入口节点**到 \(n\) 的每条路径都必须经过 \(d\)。  
特殊地，每个节点都支配自身。

- **严格支配**：\(d\) 严格支配 \(n\)（\(d \text{ sdom } n\)）若 \(d \text{ dom } n\) 且 \(d \ne n\)。
- **直接支配者（Immediate Dominator）**：节点 \(n\) 的直接支配者 \(idom(n)\) 是严格支配 \(n\) 且不严格支配任何其他严格支配 \(n\) 的节点的节点。  
- **支配树（Dominator Tree）**：以入口节点为根，边为 \(idom(n) \to n\) 构成的树。
- **支配图**：通常指支配树，有时也指包含所有支配关系的 DAG（但前者更常用）。

### 1.2 构造算法：Lengauer–Tarjan 算法

最经典的算法是 **Lengauer–Tarjan（1979）**，时间复杂度接近线性 \(O(|E| \log |V|)\) 或使用路径压缩达到 \(O(|E| \alpha(|E|,|V|))\)。

**算法步骤（简化）**：

1. 对 CFG 进行 DFS，为每个节点分配 DFS 序号（`dfn`），并按 DFS 树建立父子关系。
2. 初始化半支配者（`semi`）数组，`semi[v] = v`。
3. 按 DFS 序号**逆序**处理节点（除入口外）：
   - 对每个前驱 \(u\)：
     - 计算 \(u\) 的候选半支配者：若 `dfn[u] < dfn[v]` 则候选为 \(u\)，否则候选为 `semi[ancestor(u)]`。
     - 取所有候选中 DFS 序号最小的作为 `semi[v]`。
   - 将 \(v\) 加入 `semi[v]` 的桶中。
   - 链接 \(v\) 到其在 DFS 树中的父节点。
   - 处理父节点的桶：对桶中每个 \(w\)，通过 `ancestor(w)` 求 `idom[w]`。
4. 第二遍扫描确定最终直接支配者。

该算法高效且被广泛用于 LLVM、GCC 等编译器。

## 📌 2. 控制依赖图（CDG）

### 2.1 定义

控制依赖源于**后向支配（Post-dominance）**关系。节点 $p$ **后向支配** 节点 $n$（记作 $p \text{ pdom } n$），当且仅当从 $n$ 到**出口节点**的每条路径都必须经过 $p$。

- **控制依赖**：节点 \(n\) **控制依赖于** 边 \(e = (a \to b)\)，若：
  1. \(n\) 后向支配 \(b\)（即 \(n\) 在从 \(b\) 到出口的所有路径上），且
  2. \(n\) 不严格后向支配 \(a\)（即存在从 \(a\) 出发不经过 \(n\) 的路径到出口）。

直观理解：条件分支 \(a\) 的两个后继之一 \(b\) 导致 \(n\) 一定执行，而另一个后继可能导致 \(n\) 不执行，则 \(n\) 的执行与否由 \(a\) 处的条件决定。

**控制依赖图**的节点为 CFG 节点（或基本块），边 \(u \to v\) 表示 \(v\) 控制依赖于 \(u\) 的某个出边。

### 2.2 构造算法：基于后向支配树

经典算法由 **Ferrante, Ottenstein, Warren（1987）** 提出：

**输入**：CFG 及其后向支配树（对逆向 CFG 求支配树即可得后向支配树）。
**步骤**：

1. 构建逆向 CFG（将边反向，交换入口/出口），求其支配树即为原 CFG 的**后向支配树**。
2. 确定控制依赖边：对于 CFG 中的每条边 \(A \to B\)，设 \(L\) 为 \(A\) 和 \(B\) 在后向支配树中的**最低公共祖先（LCA）**。
3. 沿着后向支配树，从 \(B\) 向上遍历至 \(L\) 的子节点（不含 \(L\)），路径上的每个节点都**控制依赖于** \(A\)。
4. 添加边 \(A \to\) 路径上节点（或更常用：边 \(A \to\) 对应的条件分支语句）。

**伪代码**：
```
for each edge (A -> B) in CFG:
    L = LCA(A, B) in post-dominator tree
    current = B
    while current != L:
        add control dependence edge A -> current
        current = parent(current) in post-dominator tree
```

该算法复杂度 \(O(|E| \cdot h)\)，其中 \(h\) 为后向支配树高度，实际中效率很高。

## 📌 3. 数据依赖图（DDG）

### 3.1 定义

数据依赖描述**变量定义与使用**之间的关系。在程序分析中，常见三种数据依赖：

- **流依赖（Flow Dependence / Read-After-Write, RAW）**：语句 S2 使用了 S1 定义的变量，则 S2 流依赖于 S1。
- **反依赖（Anti-dependence / Write-After-Read, WAR）**：S2 重新定义了 S1 读取的变量，则 S2 反依赖于 S1。
- **输出依赖（Output Dependence / Write-After-Write, WAW）**：S2 和 S1 都定义了同一变量，则 S2 输出依赖于 S1。

在二进制分析（如 angr）中，数据依赖图通常基于 **VEX IR 语句** 构建，节点为语句实例，边表示 RAW 依赖（有时也包括 WAR/WAW，但 RAW 最核心）。

### 3.2 构造算法：到达定值（Reaching Definitions）分析

数据依赖图的构建需要**数据流分析**，核心是计算每个程序点的**到达定值集合**。

**经典数据流方程（到达定值）**：

- \( \text{gen}[B] \)：基本块 \(B\) 内定义且能到达块尾的变量集合。
- \( \text{kill}[B] \)：被 \(B\) 中定义覆盖的变量集合。
- \( \text{in}[B] = \bigcup_{P \in \text{pred}(B)} \text{out}[P] \)
- \( \text{out}[B] = \text{gen}[B] \cup (\text{in}[B] - \text{kill}[B]) \)

通过迭代求解到达定值后，即可建立**定义-使用链（DU-chain）**：
- 对每个变量的使用点，查找到达该点的所有定义，形成 RAW 依赖边。
- 同样可建立 UD-chain（使用-定义链）用于后向分析。

在二进制层面，由于寄存器重用和内存别名，分析更为复杂，常需**值集分析（VSA）** 或**符号执行**来近似。

**angr 的 DDG 构造**：
1. 基于 CFG，对每个基本块进行 VEX IR 提升。
2. 执行轻量级数据流分析（可能结合符号执行）跟踪寄存器和内存位置的 def-use。
3. 构建以 VEX 语句为节点的图，边为 RAW 依赖。

## 🔄 4. 三者区别与联系总结

| 图类型         | 依赖维度   | 节点类型       | 边的含义                                   | 关键输入         | 典型用途                     |
|----------------|------------|----------------|--------------------------------------------|------------------|------------------------------|
| **支配图**     | 控制流必经 | 基本块         | 支配关系（所有路径必经）                   | CFG + DFS        | 识别循环、SSA 构建、优化     |
| **控制依赖图** | 控制条件   | 基本块         | 条件分支决定某节点是否执行                 | CFG + 后向支配树 | 程序切片、并行化、安全分析   |
| **数据依赖图** | 数据流动   | 语句/指令      | 变量定义被使用（RAW 为主）                 | CFG + 数据流分析 | 指令调度、切片、漏洞挖掘     |

**联系**：
- **支配图是 CDG 的基础**：后向支配树由逆向 CFG 的支配树得到，而 CDG 构建直接依赖后向支配树。
- **CDG 与 DDG 共同构成程序依赖图（PDG）**：PDG 将控制依赖和数据依赖统一为图，是后向切片的直接输入。
- **支配信息辅助数据流分析**：例如利用支配边界（Dominance Frontier）放置 φ 节点以构建 SSA 形式，进而简化数据依赖计算。

通过上述三图的组合，angr 能够以 VEX IR 为粒度执行精准的程序切片，从而穿透控制流平坦化等混淆技术。

## 逆向分析的符号执行

编译器优化（如常量传播、死代码消除）中的符号执行通常是静态的、流不敏感的或路径不敏感的，旨在安全近似程序行为，不追求完整路径覆盖，但必须保证优化不改变程序语义。

[SCCP(Sparse Conditional Constant Propagation)](../../symbolic_execution/sccp/sccp.md) 是经典代表，它在 SSA 形式 上结合了常量传播和死代码消除。SCCP 本质上是一种简化的、路径不敏感的静态符号执行。这种设计在编译器优化中足够用，因为优化必须快速且安全。

逆向工程、漏洞挖掘等场景中，符号执行的目标是深入理解程序行为、自动生成输入以触发特定路径或证明路径可达性。

典型流程：

1. 将程序输入、寄存器、内存初始化为符号变量。

2. 维护路径约束（Path Constraint）：每个分支条件的符号表达式。

3. 使用 SMT 求解器（如 Z3）判断路径可行性，并可生成满足约束的具体输入。

二进制符号执行引擎的核心算法，本质上是一个带约束求解的、状态分叉的模拟器。它通过将程序输入、寄存器和内存抽象为符号，通过模拟执行并收集路径约束，来探索程序在不同输入下的行为。它将静态的 def-use 关系升级为了动态的、路径敏感的符号计算，以精确解析那些静态分析难以处理的指针、内存等复杂依赖。
